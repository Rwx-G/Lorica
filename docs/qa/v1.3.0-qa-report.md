**Author:** Romain G.
**Date:** 2026-04-12
**Release:** v1.3.0 - Authentication & advanced routing
**Method:** BMAD QA (risk profile + trace matrix + NFR assessment + per-feature gates)

## Executive Summary

v1.3.0 ships 10 features across 4 phases: connection pre-filter, cache predictor, cache Vary, header-based routing, canary traffic split, forward auth (with opt-in verdict cache), request mirroring, stale-while-revalidate (with dangling-permit safety net), response body rewriting, mTLS client verification. All features are production-complete with 100% acceptance-criteria trace coverage (65/65 AC mapped to tests), 52 end-to-end tests driving real Pingora servers against mock backends, Prometheus counters on every non-trivial feature, and zero open mitigation debt at release time.

**Release gate: PASS.** All 10 feature gates at 98/100. No CONCERNS. No backlog.

## Schema Migrations

v1.3.0 introduces 7 schema migrations, all idempotent `ALTER TABLE routes ADD COLUMN`:

| Version | Column | Feature |
|---------|--------|---------|
| V25 | `cache_vary_headers TEXT NOT NULL DEFAULT '[]'` | Cache Vary |
| V26 | `header_rules TEXT NOT NULL DEFAULT '[]'` | Header routing |
| V27 | `traffic_splits TEXT NOT NULL DEFAULT '[]'` | Canary |
| V28 | `forward_auth TEXT DEFAULT NULL` | Forward auth (carries `verdict_cache_ttl_ms` in JSON) |
| V29 | `mirror TEXT DEFAULT NULL` | Mirroring |
| V30 | `response_rewrite TEXT DEFAULT NULL` | Response rewrite |

## Observability (Prometheus)

v1.3.0 adds 5 per-route counters (all with `route_id` label and bounded cardinality):

| Metric | Labels | Source |
|--------|--------|--------|
| `lorica_cache_predictor_bypass_total` | `route_id` | cache predictor |
| `lorica_header_rule_match_total` | `route_id`, `rule_index` | header routing |
| `lorica_canary_split_selected_total` | `route_id`, `split_name` | canary |
| `lorica_mirror_outcome_total` | `route_id`, `outcome` | mirroring |
| `lorica_forward_auth_cache_total` | `route_id`, `outcome` | forward auth |

All are registered in `lorica-api::metrics::REGISTRY` and exposed via the existing `GET /metrics` endpoint.
