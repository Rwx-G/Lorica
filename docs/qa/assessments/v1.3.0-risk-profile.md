# Risk Profile - v1.3.0 (closed)

**Author:** Romain G.
**Date:** 2026-04-12
**Scope:** v1.3.0 release (10 features)
**Method:** BMAD risk-profile (probability × impact, 1-3 scale)

This profile is the *closed* version of the initial risk register.
Every risk that had mitigation debt has been resolved in-session
before shipping; the register below documents residual risk only.

---

## Risk Scoring

- **Probability:** 1 = rare (requires misconfig + edge trigger), 2 = plausible (happens under normal operator mistakes), 3 = likely (will happen in prod without mitigation)
- **Impact:** 1 = low (log noise, UX glitch), 2 = medium (one route degraded), 3 = high (data leak, outage, bypassed security)
- **Score = P × I.** 6 = critical, 4-5 = high, 2-3 = medium, 1 = low.

## Residual risks (by-design, operator-controlled)

| ID | Feature | Risk | P | I | Score | Category | Notes |
|----|---------|------|---|---|-------|----------|-------|
| R-RR-03 | Response rewrite | Regex with catastrophic backtracking freezes worker | 1 | 3 | 3 | reliability | Rust `regex` crate uses linear-time NFA; the class of ReDoS attack does not apply. Validator pre-compiles at write time |
| R-FA-02 | Forward auth | Auth allowlist of injected response headers includes a privileged header the operator didn't anticipate | 2 | 2 | 4 | security | Validator checks non-empty names; operators must curate the list. Documented on `ForwardAuthConfig.response_headers` |
| R-CANARY-02 | Canary | Client IP absent (Unix socket, rare IPv6) → request falls back to route defaults even though operator expected a canary | 2 | 1 | 2 | ux | Documented on `TrafficSplit` |
| R-CONN-01 | Connection filter | CIDR parse error on reload → filter falls back to previous policy (warn log) | 1 | 3 | 3 | security | Defensive fallback prevents opening the gate on operator typo |
| R-CONN-02 | Connection filter | Default-deny flag flipped without allow list → all connections dropped | 2 | 3 | 6 | availability | **By design** (opt-in via non-empty allow list switches to default-deny). Dashboard shows allow/deny counts |
| R-CACHE-01 | Cache Vary | Route `cache_vary_headers` explosion (50+ headers) → unbounded cache cardinality | 2 | 2 | 4 | perf | `Vary: *` fallback anchors on URI; hard cap not implemented. Operator judgment required |
| R-CACHE-02 | Cache predictor | LRU false positive marks a transient-uncacheable asset as permanently bypassed | 1 | 1 | 1 | perf | Only *OriginNotCache*, *ResponseTooLarge*, custom reasons are remembered - not transient errors |
| R-SWR-02 | SWR | Thundering herd during SWR window if cache_lock isn't held | 1 | 2 | 2 | perf | `stale_writer` in cache_lock module handles this; tested by swr_e2e_test::concurrent_requests_coalesce_on_single_refresh |
