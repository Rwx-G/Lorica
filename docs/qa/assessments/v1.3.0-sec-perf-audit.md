# v1.3.0 Security + Performance Audit

**Author:** Romain G.
**Date:** 2026-04-12
**Scope:** All v1.3.0 feature code after the initial QA closeout (commit fc559c2).

This audit focused on the newly-added code paths. Feature-level design and AC coverage were already validated in the main QA report; this pass looks for residual bugs in attack surface or hot-path cost.

---

## Findings (all fixed in this pass)

### SEC-AUD-01 - Verdict cache 64-bit hash collision (MEDIUM, fixed)

**Symptom:** The forward-auth verdict cache keyed on a 64-bit `DefaultHasher` output of `(route_id, cookie)`. Birthday-attack cost to find two keys hashing to the same u64 is ~2^32 distinct sessions, which is feasible on a large multi-tenant deployment. Because DashMap uses the provided `String` key as the lookup identity, two colliding keys would actually resolve to the same cache entry - meaning user B could have received user A's cached `Allow` response_headers (e.g. `Remote-User: alice`).

**Fix:** `verdict_cache_key` now returns the literal NUL-separated concatenation `"{route_id}\0{cookie}"` instead of a truncated hash. DashMap's internal sharding still hashes for bucket selection, but *lookup* uses full `String` equality, so two distinct raw keys can never match the same entry. Memory cost is bounded by the existing 16 384-entry cap: at most ~8 MiB for realistic cookie sizes, which is already comparable to what the value side (`response_headers`) holds.

**Regression guards:**
- `test_verdict_cache_key_concatenates_with_nul_separator` - asserts the key shape (boundary + prefix).
- Existing `test_verdict_cache_key_varies_by_route` / `_by_cookie` / `_stable_for_same_inputs` continue to pass.

### PERF-AUD-01 - Verdict cache O(n) eviction under flood (MEDIUM, fixed)

**Symptom:** On cap overflow the previous code iterated the DashMap and removed 25% of entries:

```rust
let victims: Vec<String> = FORWARD_AUTH_VERDICT_CACHE
    .iter().take(target).map(|e| e.key().clone()).collect();
for v in victims { FORWARD_AUTH_VERDICT_CACHE.remove(&v); }
```

Each call was O(n + k) with per-shard locks, plus 4 096 allocations for the victim `Vec`. Under a cookie-flood attack (attacker sends a stream of requests with distinct session cookies), every insert past the 16 384-entry cap triggered this O(n) scan - sustaining ~0.5 ms of DashMap work per hostile request. At 1 000 req/s sustained that is ~50% of one core spent on eviction bookkeeping.

**Fix:** Bounded FIFO via a sibling `parking_lot::Mutex<VecDeque<String>>` that records insertion order. `verdict_cache_insert` now does:

```rust
while order.len() >= VERDICT_CACHE_MAX_ENTRIES {
    if let Some(old) = order.pop_front() { CACHE.remove(&old); } else { break; }
}
order.push_back(key.clone());
CACHE.insert(key, value);
```

O(1) amortised per insert. Under flood each insert pops exactly one old entry.

**Regression guards:**
- `test_verdict_cache_reset_for_test_clears_both_containers` - exercises the insert path and verifies queue+map stay in lockstep. A big-insert stress test was explored but removed because it shares global state with other tokio tests and flakes under cargo's default parallel execution; the FIFO invariant is straightforward inspection of the insert body.

### SEC-AUD-02 - Mirror forwards credentials to shadow (LOW, documented)

**Symptom:** `build_mirror_forward_headers` forwards every request header (including `Cookie` and `Authorization`) to the shadow backend. This is industry-standard (Nginx `mirror`, Traefik `Mirroring`, Envoy `request_mirror_policies`) because shadow testing needs the same authentication context as the primary. But an operator could deploy a shadow outside the primary's trust boundary and leak session tokens.

**Fix:** Documented the trust model loudly in `MirrorConfig` doc comment: shadow backends must be in the same trust boundary as the primary. Not a code fix - adding a "strip credentials" knob would defeat the purpose of shadow testing by default. The loud doc is the right lever.

## Non-findings (verified clean)

| Area | Verification |
|------|--------------|
| mTLS `compute_ca_fingerprint` uses 64-bit hash | Fingerprint is for drift *detection*, not security. Inputs come from the operator's DB, no attacker-controlled values. 64-bit collision probability negligible for the drift-detection use case |
| Response rewrite regex DoS | Rust `regex` crate is linear-time NFA by construction; ReDoS class attacks do not apply. Write-time compilation catches malformed patterns |
| Connection filter reload parse DoS | Parse errors fall back to previous policy via ArcSwap; cannot open the gate on operator typo |
| Prometheus label cardinality | All labels are either fixed-set strings (outcome, rule_index stringified from small integers) or bounded by route/backend count (operator config). No user-input-derived labels |
| Mirror 256-slot semaphore + body cap | `try_acquire_owned` drops on saturation (no queue backlog). Body capped at `max_body_bytes` (≤128 MiB); worst-case 32 GiB memory under full saturation is operator-controlled via cap |
| Forward auth header forwarding to auth service | Same trust model as mirror: Authelia must be in trust boundary. Documented |
| SWR LockCtx Drop safety net | Verified LockCtx::Drop releases the inner WritePermit via explicit Option::take; no double-release. Refresh-failure path now deterministic in debug + release |
