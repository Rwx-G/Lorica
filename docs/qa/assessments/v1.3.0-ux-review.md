## Review Axes

The review scores each surface on 7 UX dimensions. For each finding a severity is assigned using real user impact, not aesthetic preference:

- **SEV-1** - blocks a task, causes data loss risk, fails accessibility baseline. Fix before ship.
- **SEV-2** - friction that slows the user or forces a round-trip to the API for error resolution. Fix soon.
- **SEV-3** - consistency / polish. Fix when convenient.

## Findings

### F-01 - SEV-2 - Tab overflow: 11 tabs, no visual affordance for scroll

**Where:** `RouteDrawer.svelte` lines 335-342. `.tab-bar` uses `overflow-x: auto` + `flex-shrink: 0` on tab buttons.

**Problem:** v1.3.0 grew the tab count from 8 to 11 (added Header Rules, Canary, Rewrite). On a 1280-wide drawer with default labels the last two tabs now wrap off the right edge on typical 13-inch laptop displays. The tab bar *is* scrollable horizontally, but there is no scroll-hint gradient / shadow / arrow. Users with routes that have header rules or response rewrites may never find those tabs.

**Fix (suggested):**
- Add a soft right-edge gradient when content overflows (`mask-image: linear-gradient(to right, black 90%, transparent)`) so users see there's more.
- OR: group related tabs into one "Routing" group (header_rules + canary + path_rules) collapsed under a dropdown tab.

### F-02 - SEV-2 - Validation errors aren't localized to the offending field

**Where:** `RouteDrawer.svelte` line 173-175 renders `formError` as a single banner at the drawer top. `validateRouteForm` in `route-form.ts` returns a single string like "Response rewrite rule 3: invalid regex (unclosed group)".

**Problem:** With features like Header Rules / Canary / Response Rewrite that iterate arrays of rules, an error at "rule #3" requires the user to manually count down cards to find the culprit. On a scrolled form where rule #3 is collapsed, the user has no indication which card to expand.

**Fix (suggested):**
- Extend `validateRouteForm` to return `{ message, field_path }` where `field_path` is e.g. `"header_rules[2].value"`.
- In the tab component, when an error refers to an indexed rule, auto-expand that index and scroll it into view.
- A minimal version: just auto-switch to the correct tab on validation error (already inferrable from `TAB_FIELDS`).

### F-03 - SEV-2 - `SecurityTab` subsection overload (4 features in one tab)

**Where:** `SecurityTab.svelte` now hosts: security-headers preset, max-body, rate-limit, ip-allowlist, ip-denylist, basic-auth, forward-auth, mirror, mTLS - 9 subsections totalling ~700 lines of DOM on render.

**Problem:** User task "configure mTLS" becomes scroll-and-find rather than pick-and-fill. A dot badge on the "Security" tab tells the user *some* field in Security was modified, but can't tell them *which* subsection.

**Fix (suggested):**
- Option A (minimal): add a sticky table-of-contents rail inside the tab with anchors to each subsection.
- Option B (better): split into "Security" (headers, max-body, rate-limit, ip-lists) and "Authentication" (basic-auth, forward-auth, mTLS) + keep Mirror in... "Security" is actually the wrong home for mirror anyway - it's a routing/observability feature. Consider moving Mirror next to Canary or into its own "Observability" tab.

### F-04 - SEV-3 - Input-pattern inconsistency: CSV vs newline-separated lists

**Where:** mixed across the dashboard:

| Field | Separator | Where |
|-------|-----------|-------|
| `cache_vary_headers` | CSV | CachingTab |
| `forward_auth_response_headers` | CSV | SecurityTab |
| `mtls_allowed_organizations` | CSV | SecurityTab |
| `ip_allowlist`, `ip_denylist` | newlines | SecurityTab (pre-v1.3.0) |
| `connection_allow_cidrs`, `connection_deny_cidrs` | newlines | Settings |
| `hostname_aliases` | CSV | GeneralTab |

**Problem:** The distinction is not documented. Users learn by trying. For CIDRs the newline choice is defensible (CIDRs don't contain commas either, but whitespace tolerance matters on paste). For header names / org names, CSV is defensible (short tokens, typical config file convention). Still - inconsistency costs the user one cognitive beat per new feature.

**Fix (suggested):**
- Leave as-is but document the convention in the "Help" / tooltips: "Comma-separated" vs "One per line". Current hint text does this inconsistently.
- Or: standardize all "list of tokens" inputs to accept *both* separators (split on `[,\n]+`). Low effort, removes the footgun.

### F-05 - SEV-3 - Disabled inputs without visible "why"

**Where:** multiple:
- `forward_auth_timeout_ms`, `forward_auth_response_headers` disabled when `forward_auth_address` is empty.
- `mirror_sample_percent`, `mirror_timeout_ms`, `mirror_max_body_bytes` disabled when `mirror_backend_ids` is empty.
- `mtls_required`, `mtls_allowed_organizations` disabled when `mtls_ca_cert_pem` is empty.

**Problem:** Users who didn't read the hint below may wonder why the field is greyed out. The pattern is correct (progressive disclosure), but lacks the "enable me by doing X" cue.

**Fix (suggested):**
- Add `title="Set {parent field} first to enable this option"` on every disabled input.
- Or: render a subtle "Enable mTLS by pasting a CA PEM above" placeholder in the disabled inputs when empty.

---

### F-06 - SEV-2 - Rule-removal has no confirm, no undo

**Where:** `HeaderRulesTab.svelte` line 30 (`removeRule`), `CanaryTab.svelte` (same), `ResponseRewriteTab.svelte` (same). All are one-click destructive with no dialog or recovery.

**Problem:** Misclicking the trash icon on a 20-rule Response Rewrite config deletes the rule with no recovery short of closing-without-saving (which loses all other edits). The icons sit next to move-up / move-down / expand buttons - a real misclick risk.

**Fix (suggested):**
- Minimum: `if (!confirm('Remove this rule?')) return;` — 1 line, brutal but safe.
- Better: change the trash icon to a two-step "X → confirm" inline button (first click arms, second click removes, 3s revert window), matching Gmail / Notion patterns.
- Better still: soft-delete with a bottom toast "Rule removed · Undo" for 5s.

### F-07 - SEV-3 - `ResponseRewriteTab` shows global settings even when no rules exist

**Where:** `ResponseRewriteTab.svelte` lines 83-105 render `response_rewrite_max_body_bytes` + `response_rewrite_content_type_prefixes` unconditionally, even when `rules.length === 0`.

**Problem:** New user lands on the tab, sees two configured knobs before any rule exists. Order of operations is inverted from "enable by adding a rule, then tune."

**Fix (suggested):** Hide the global settings block until at least one rule exists, or render it *below* the rules list with a dim visual state until the feature is active.

---

### F-08 - SEV-1 - `canary weight_percent` input has no `label-for`

**Where:** `CanaryTab.svelte` lines 112-118. The `<input type="number">` for weight lives next to a visual `%` suffix but no associated `<label>`.

**Problem:** Screen readers announce "edit text, blank" for the weight input. The visual context (weight label on the row) is only announced if the user navigates back to it first. Fails WCAG 1.3.1 (info and relationships).

**Fix:** Add `<label for="canary-weight-{index}" class="sr-only">Weight percent</label>` and matching `id` on the input. Or use `aria-label="Weight percent"`. Minor DOM change.

---

### F-09 - SEV-2 - Mirror tab lacks feedback parity with Canary

**Where:** `CanaryTab.svelte` displays a weight-summary bar (total allocated %, warning if > 100). `SecurityTab.svelte` Mirror subsection has no equivalent summary.

**Problem:** The operator configuring multiple mirror backends loses the at-a-glance count. Under saturation they'd want to know "3 shadows × 100% sample = 3 concurrent mirror sub-requests per primary request".

**Fix (suggested):** Add a one-line summary like "3 shadow backends · sampling 100% · max body 1 MiB" below the backend checkbox list. Cheap, follows the pattern established by Canary.

---

### F-10 - SEV-3 - Inconsistent "+ Add" button labels

**Where:**
- HeaderRulesTab: `+ Add header rule`
- CanaryTab: `+ Add traffic split` (assumed; needs verification)
- ResponseRewriteTab: `+ Add rule` (assumed)
- PathRulesTab (pre-v1.3.0): `+ Add path rule`

**Problem:** Minor copy drift. Fine functionally, but "add rule" vs "add X rule" on otherwise-identical accordion patterns is grist for "seems inconsistent" feedback in user testing.

**Fix (suggested):** Standardize to `+ Add {noun}` where `{noun}` matches the section: "+ Add header rule", "+ Add traffic split", "+ Add rewrite rule", "+ Add path rule".

---

### F-11 - SEV-3 - mTLS restart-required banner could be more actionable

**Where:** `SecurityTab.svelte` lines 255-264. Banner appears when `form.mtls_ca_cert_pem !== initialMtlsCaCertPem`.

**Problem:** Banner copy explains the why but doesn't tell the user *how* to restart. On a systemd deployment it's `systemctl restart lorica`; on Docker it's `docker restart lorica`; in-container it's a different dance. Users unfamiliar with their own deployment might ask support.

**Fix (suggested):** Add one line "On systemd: `systemctl restart lorica` - on Docker: `docker restart <container>`" OR a link to a specific docs section. Current copy is accurate but not actionable.

---

### F-12 - SEV-3 - Disabled-state contrast may fail WCAG AA

**Where:** `SecurityTab.svelte` line 329 and mirrored across tabs:
```css
.form-group input:disabled { opacity: 0.5; cursor: not-allowed; }
```

**Problem:** Muted text at 0.5 opacity on the default muted-text color (~#888) lands at ~3:1 contrast on the light theme background - that's AA for large text only, fails for body-size inputs. Dark theme needs separate check.

**Fix (suggested):** Use `color: var(--color-text-muted); background: var(--color-input-disabled-bg);` with a dedicated disabled background token tuned to pass 4.5:1. Alternatively, don't rely on opacity alone - pair with the "why disabled" tooltip from F-05.

---

### F-13 - SEV-2 - `mtls_required` checkbox lacks a unique `id` / `label for`

**Where:** `SecurityTab.svelte` lines 269-281. Checkbox wrapped in `.checkbox-item` with visible text "Required (reject if no client cert)".

**Problem:** The `<label>` wraps the input (implicit association) which is technically OK for a11y, but there's no `id` on the input so other a11y tooling (form-field-error associations, focus targets from validation messages) can't reference it.

**Fix (suggested):** Add `id="mtls-required"` on the input. One-line change.

---

### F-14 - SEV-3 - No "test connection" / "validate PEM" action

**Where:** Forward auth URL, mTLS CA PEM. Users paste config and only discover it's wrong after saving and sending a real request.

**Problem:** Round-trip cost for a typo: save (maybe wait for rollout) → send a request → observe failure → read logs → correct. That's minutes per typo on a production-shaped workflow.

**Fix (suggested):**
- Forward auth: add a "Test" button that sends a `GET {address}` from the dashboard backend with a dummy session, shows status + first headers.
- mTLS: add a "Validate PEM" button that calls the existing API validator (already does X.509 parsing) and surfaces the summary ("Trusted 2 CAs: `Acme CA`, `Beta CA`").
- Both: out of scope for a pure UX review fix, but flagged because it's real user pain.

## Non-findings (things that looked suspicious but check out)

- **Imported / modified markers:** consistent placement across all v1.3.0 tabs; users can tell at a glance which fields came from a TOML import vs operator edit.
- **Empty states:** every new tab has clear empty-state copy explaining the feature. Good.
- **Placeholders:** realistic values (`X-Tenant`, `acme`, `http://authelia.internal:9091/api/verify`) that teach the format by example.
- **Canary weight summary:** great live-feedback pattern. Set the bar for others to follow (see F-09).
- **mTLS restart banner:** correct detection (only fires on real PEM change, not on cosmetic whitespace). Refinement suggested (F-11) but the core behaviour is right.
- **Header rule "disabled" badge:** clear visual, good tooltip copy. No change needed.
