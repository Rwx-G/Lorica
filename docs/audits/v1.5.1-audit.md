## Highs (5)

### H-1 - `bot_hmac_secret_hex` leaks via `GET /api/v1/settings` AND TOML export

- **Severity:** High
- **Files:**
  - `lorica-api/src/settings.rs:14-20` - `get_settings` returns raw `GlobalSettings` JSON, no scrub
  - `lorica-config/src/export.rs:43-95` - export does not redact the field
  - `lorica-config/src/models/settings.rs:275-287` - field's own doc claims it is "scrubbed from `GET /api/v1/settings` responses at the API layer"
- **Impact:** Authenticated admin (anyone holding a session cookie) can read the 32-byte HMAC secret used to sign bot-protection verdict cookies. With that secret an attacker forges a "human" verdict for every IP, bypassing the bot challenge across every route until the next certificate renewal rotates the secret. Field is also persisted into TOML exports - any historical export shipped to CI / git / S3 carries the secret.
- **Fix:** In `get_settings`, deep-clone the settings struct and overwrite `bot_hmac_secret_hex` with `String::new()` (or a `**REDACTED**` sentinel) before serialising. Mirror the same scrub in `export.rs::export_to_toml`. Add an `import.rs` validator that rejects the literal `**REDACTED**` value (analogous to the existing `password_hash` / `key_pem` guards) so a round-trip cannot accidentally clear a live secret.
- **Effort:** 30-line change, 1 unit test pinning the redaction.

### H-2 - WAF body scan silently truncates past 1 MiB - bypass via padding

- **Severity:** High
- **File:** `lorica/src/proxy_wiring.rs:3768-3808` (`WAF_BODY_SCAN_MAX = 1_048_576`)
- **Evidence:** Body bytes past the 1 MiB cap are NOT buffered for WAF scanning, but ARE forwarded upstream. No companion 413 / WAF-block path when the body exceeds the scan cap.
- **Impact:** Attacker prefixes 1 MiB of inert padding (e.g. `<!-- ... -->` HTML comment, `null,null,null...` JSON noise) to any malicious payload. Padding fills the buffer; malicious tail is never seen by `evaluate_body`; upstream gets the full request including unscanned attack. Effective on every route where `max_request_body_bytes` is unset or larger than 1 MiB.
- **Fix:** Either (a) when the buffer hits the cap, stop forwarding and return 413 with "request body exceeds WAF scan window" - safest default, (b) flip the route into fail-closed posture (block in Blocking mode, mark request "WAF unverified" header in Detection mode), or (c) raise the cap to match the route's `max_request_body_bytes` and document the memory cost. Current "scan a window, forward the rest" is the worst of the three.

### H-3 - Three CRS-style WAF rules cannot fire (header-name-in-value pattern)

- **Severity:** High (reduces protection silently; advertised but inert)
- **File:** `lorica-waf/src/rules.rs:269-282, 416-425, 533-541` (rules 920100 / 920120 / 920140)
- **Evidence:** Patterns expect literal header NAME (`content-length:`, `transfer-encoding:`, `X-Forwarded-Host:`) inside the SCANNED VALUE. But `lorica/src/proxy_wiring.rs:3573-3588` calls `waf_engine.evaluate(...)` with `headers: &[(&str, &str)]` already split by Pingora's parser. A real `Transfer-Encoding: chunked, chunked` arrives as the tuple `("transfer-encoding", "chunked, chunked")` - the value never contains `transfer-encoding:`.
- **Impact:** Three named protections are inert. CRS 920140 is the canonical CL/TE smuggling guard. Rule appears in the listing UI - operator believes the proxy ships smuggling protection that does not exist. Pingora's strict httparse already rejects most CL/TE shapes at the parser level so practical exposure is bounded, but the inertness itself is the issue.
- **Fix:** Either rewrite patterns to match plain values (e.g. for 920140 detect `chunked.*chunked` or `,` in the `transfer-encoding` value alone) OR move the smuggling check into a dedicated header-name-aware predicate that runs before the regex pass on the `(name, value)` tuples. Confirm via `tests-e2e-docker` that a `Transfer-Encoding: chunked, chunked` request now hits the rule in Detection mode.

### H-4 - WAF URL decoding maps bytes via `byte as char` (latin-1 mojibake), defeating UTF-8 attack signatures

- **Severity:** High (cross-flagged by both manual security audit and manual code-quality audit)
- **File:** `lorica-waf/src/engine/url_decode.rs:54`
- **Evidence:**
  ```rust
  if let Ok(byte) = u8::from_str_radix(&hex, 16) {
      result.push(byte as char);  // <-- u8 -> Unicode codepoint -> 1-2 UTF-8 bytes
      continue;
  }
  ```
  `byte as char` casts `u8` to `U+0000..=U+00FF`, then `String::push` re-encodes as 1-2 UTF-8 bytes. So `%C0%BC` (overlong `<` for classic WAF bypasses) decodes to the codepoints `U+00C0 U+00BC`, not the byte sequence `0xC0 0xBC`. Multi-byte UTF-8 like `%C3%A9` (`é`) decodes to `U+00C3 U+00A9` instead of `U+00E9`. Downstream SQLi/XSS regexes never see the actual attack bytes. Also: `+` -> space transform applied to ALL fields including paths/headers, but per RFC 3986 only `application/x-www-form-urlencoded` (forms in query/body) treats `+` as space.
- **Test gap:** `engine/tests.rs:269..292` only covers ASCII. No test for byte ≥ 0x80, no test for `%C0%80` overlong NULL, no test for `%00`.
- **Fix:** Decode into `Vec<u8>` and finalise via `String::from_utf8_lossy(&bytes).into_owned()`. Skip the `+` -> space transform for path/headers; keep only for `query`. Tests: a `é` URL-encoded body must still match a literal `é` rule; a header value `a+b` must NOT trip ` or 1=`; `%C0%80` must not slip through.

### H-5 - Story 7.3 settings missing in code: `header_timeout_s` and `flood_strict_rps`

- **Severity:** High (acceptance-criteria violation on a shipped story; operator-visible contract gap)
- **Files:**
  - `docs/stories/story-7.3-anti-ddos.md:32` checked `[x]`: `Add configurable header_timeout_s field to global proxy settings (default 10s)`
  - `docs/stories/story-7.3-anti-ddos.md:40` checked `[x]`: `Add configurable flood_threshold_rps and flood_strict_rps fields to global settings`
  - `lorica-api/src/settings.rs` declares `flood_threshold_rps`, `waf_ban_threshold`, `waf_ban_duration_s` only. **Workspace-wide grep for `header_timeout_s` and `flood_strict_rps` returns matches only inside the story md.**
- **Impact:** Slowloris detection (AC #1: "configurable threshold") is using a hardcoded constant or Pingora's default header read timeout. Same for the flood-strict mode fall-down (AC #5): without `flood_strict_rps` the "stricter" target is hardcoded. Promise of operator-tunable thresholds is not honoured.
- **Fix:** Either (a) wire the two settings through `settings.rs`, the per-worker config, and the proxy hot path - or (b) update the story to mark these as "implemented with non-tunable defaults" and document the chosen values. Option (a) is cleaner because slowloris timing is environment-dependent.

## Mediums (16)

### M-1 - systemd unit missing the modern hardening triad

- **File:** `dist/lorica.service` (and the byte-identical copy in `dist/lorica_1.5.1_amd64/lib/systemd/system/lorica.service`)
- Current unit has the v1 hardening (`ProtectSystem=strict`, `MemoryDenyWriteExecute=yes`, `SystemCallFilter=@system-service`, `LockPersonality=yes`). **Missing:** `ProtectClock=yes`, `ProtectHostname=yes`, `ProtectProc=invisible` + `ProcSubset=pid`, `RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX AF_NETLINK`, `RemoveIPC=yes` (verify `lorica-shmem` shutdown first), `PrivateDevices=yes`, `KeyringMode=private`, `SystemCallFilter=~@privileged @resources` to tighten `@system-service`. `PrivateUsers=yes` is incompatible with `CAP_NET_BIND_SERVICE` so stays opt-in.
- **Fix:** Add to `dist/lorica.service`, regenerate .deb/.rpm, run e2e Docker suite. Update `docs/security/hardening-guide.md`.

### M-2 - OCSP staples never refresh at runtime

- **Files:** `lorica-tls/src/ocsp.rs:147-159`, `lorica/src/reload.rs:775-799`
- `try_fetch_ocsp` only invoked from `reload_cert_resolver` (config reload / cert install). No background refresh loop, no `ocsp_refresh_interval` setting, no advisory pin against the response's `nextUpdate`.
- **Impact:** A Lorica process running 30+ days without a reload happily continues to staple a 3-week-old "good" response. Most clients soft-fail; a `Must-Staple` cert (RFC 7633) plus a strict client rejects.
- **Fix:** Spawn a `task_tracker.spawn` loop that walks the cert resolver's entries every `min(nextUpdate - now, 6h)` and re-fetches; swap each `CertifiedKey.ocsp` field via a new `CertResolver::refresh_staples(...)` that builds a fresh `Inner` and `arc-swap`s it. Surface `lorica_ocsp_refresh_total{result="ok|fail"}`.

### M-3 - No structured admin audit log for management mutations

- **Files:** ~30 mutating handlers in `lorica-api/src/`. Only TWO `tracing::warn!` with operator context exist (`certificates.rs:563-569` for cert download; password change does NOT log).
- **Impact:** Forensics gap. Operator cannot answer "who deleted route X at 03:14?". `Session.username` is in `req.extensions()` (per `middleware/auth.rs:343-346`) but handlers ignore it.
- **Fix:** Thin `audit_log!(state, action = "route.delete", target_id = id, before = ..., session = &session)` macro feeding a dedicated structured-log target. Apply uniformly to every state-mutating handler. Tamper-resistance via hash chain is a v1.7.0 candidate.

### M-4 - Custom WAF rules silently disable the prefilter contract

- **File:** `lorica-waf/src/engine/eval.rs:55-78, 167-170`
- When ANY custom rule exists, the prefilter short-circuit is disabled for ALL fields - falling through to a full `O(rules x fields)` regex scan on every request.
- **Impact:** A trivially-broad custom rule (e.g. `\b(test)\b`) doubles or triples WAF latency on every clean request - DoS amplifier triggered by an over-eager admin.
- **Fix:** Run the prefilter unconditionally for the built-in rule pass; scan custom rules independently (no prefilter required). Unit test: "loading a custom rule must not regress prefilter behaviour on built-ins".

### M-5 - Management API serves plaintext HTTP - `Secure` cookie depends on browser-loopback exemption

- **Files:** `lorica-api/src/server.rs:598-615`, `lorica-api/src/middleware/auth.rs:315-322`
- Session cookie flagged `Secure` but API served over plaintext HTTP. Browsers treat `localhost` as "potentially trustworthy" so the cookie does flow. The moment the operator forwards via a corporate reverse proxy without TLS termination in front, the cookie silently stops being sent and the operator gets an opaque "not logged in" UX.
- **Fix:** Either (a) ship a self-signed cert at startup and serve TLS on the management port (axum + `axum-server` with rustls config), or (b) detect `Host:` headers != `localhost` / `127.0.0.1` / `::1` and reject with 421 + warn-log so the misconfig is visible. Option (a) is the cleaner ship.

### M-6 - No global per-IP TCP connection cap; only per-route `max_connections`

- **Files:** `lorica/src/proxy_wiring.rs:3410-3430`, no `max_connections_per_ip` field in settings
- A single source IP can open `max_global_connections` worth of connections to one route. Naive flooder from a small IP set DoS-es a route while staying under the global cap.
- **Fix:** Add `connection_limits_per_ip: Option<u32>` global setting. In the existing connect-time CIDR filter, increment / decrement a `DashMap<IpAddr, AtomicU32>` and refuse the accept past the cap. Defense in depth - the IP blocklist + WAF auto-ban already deflect most volumetric attacks.

### M-7 - Sync SQLite calls held across `.await` in async API handlers (executor-stall risk)

- **Files:** `lorica-api/src/logs.rs:181-184, 331-333, 451-453`, `lorica-api/src/waf.rs:67-70, 108-118`, `lorica-api/src/settings.rs:664-672`
- Every `LogStore` method is sync (`pub fn query(...)`, `lorica-api/src/log_store.rs:155`) and acquires `parking_lot::Mutex<rusqlite::Connection>`. Called inside `pub async fn get_logs(...)`, blocks the tokio worker thread. With WAL + `busy_timeout=5000`, contended write can stall a tokio worker for **up to 5 seconds**. `query_export` allows up to 100k rows - tens to hundreds of ms of stall.
- **Fix:** Wrap each call in `tokio::task::spawn_blocking({ let store = state.log_store.clone(); move || store.query(&params) }).await`. `lorica-api/src/acme/store.rs:120-132` already uses this pattern correctly.

### M-8 - Proxy hot path persists access logs synchronously per request

- **File:** `lorica/src/proxy_wiring.rs:5144-5147` (also `:2375 / :3615 / :3736 / :3904 / :3948` for WAF events)
- Every request executes a synchronous `INSERT` against `Mutex<Connection>` from inside `async fn logging`. SQLite write throughput tops at 5-10 k commits/s. With N proxy worker threads contending on one mutex, sustained throughput is limited and tail latency degrades.
- **Fix:** Decouple via a bounded `tokio::sync::mpsc::channel<LogEntry>`. Hot path does `tx.try_send(entry)` (drops on overflow with a counter); a single background task drains and batches `INSERT` (e.g. 100 rows per transaction) using `spawn_blocking`. Same channel can feed Prometheus + log-buffer subscribers.

### M-9 - `cert_export::export_from_store` blocks the global store mutex during disk IO

- **Files:** `lorica-api/src/acme/dns01.rs:357-360` and `lorica-api/src/acme/http01.rs:287-290`
- Both call sites inside `state.store.lock().await`; `export_from_store` runs `std::fs::create_dir_all` + atomic write + `chown` + `chmod` synchronously. The `tokio::sync::Mutex<ConfigStore>` is the **single serialising lock** for every API handler. Holding it across `chown + chmod + atomic rename` (worse on cross-mount EXDEV fallback) head-of-line-blocks every other handler. `reexport_all` already uses `spawn_blocking` correctly (`cert_export.rs:685`); per-cert event path bypasses that.
- **Fix:** Copy out cert + settings + ACL list while holding the lock, drop the lock, then call a refactored `export_certificate(&settings, &acls, &cert)` from `spawn_blocking`.

### M-10 - `health::http_probe` rebuilds reqwest `Client` per backend probe

- **File:** `lorica/src/health.rs:245-255`
- `reqwest::Client::builder()...build()` runs inside `async fn http_probe`, called per backend per probe interval (default 5 s). Each build allocates a new connection pool, TLS context, resolver state. With 100 backends and 5 s interval, ~20 client builds/s plus connection churn.
- **Fix:** Hoist into `static HEALTH_HTTP_CLIENT: Lazy<reqwest::Client>` (mirrors `proxy_wiring/forward_auth.rs:29` and `mirror_rewrite.rs:37`).

### M-11 - Frontend route-detail loaders have classic SPA race condition

- **Files:** `lorica-dashboard/frontend/src/routes/Sla.svelte:62-76` (`selectRoute`), pattern likely repeated in `Routes.svelte:90-...`, `Probes.svelte`, `Certificates.svelte`
- User clicks routeA, then routeB before A's `Promise.all` resolves. routeA's responses arrive after routeB's selection and overwrite the UI with stale data. The 30 s `refreshTimer` (`Sla.svelte:56`) compounds the issue.
- **Fix:** Capture `routeId` before `await` and gate every assignment: `const captured = routeId; const [...] = await Promise.all([...]); if (selectedRouteId !== captured) return;`. Or use `AbortController` + pass `signal` to `fetch`.

### M-12 - `ApiError::Internal(format!("...: {e}"))` leaks raw underlying error strings to operators

- **Files:** `lorica-api/src/auth.rs:100,197,247`, `lorica-api/src/logs.rs:183,333,392,453`, `lorica-api/src/loadtest.rs:43,102,143,...` (15 sites)
- `rusqlite::Error` can carry SQL fragments, file paths (`/var/lib/lorica/...`), library version strings - all surfaced verbatim into the dashboard toast.
- **Fix:** Log the full error via `tracing::error!` (with the request_id for correlation) and return `ApiError::Internal("internal error".into())`. Optionally include an opaque correlation token (`format!("internal error (ref {request_id})")`).

### M-13 - WebSocket reconnect timer leaks across navigation (LoadTest)

- **File:** `lorica-dashboard/frontend/src/routes/LoadTest.svelte:101-104`
- `ws.onclose = () => { wsConn = null; setTimeout(connectWS, 3000); };` while `onDestroy` only closes the current connection, not the pending reconnect. Closing the route during the 3 s backoff window leaves the timer alive; it fires after destroy and accumulates a zombie connection per cycle.
- **Fix:** Store `reconnectTimer = setTimeout(connectWS, 3000)` and `clearTimeout(reconnectTimer)` in `onDestroy`. Gate `connectWS` on a `mounted` flag.

### M-14 - OpenAPI spec missing 12 implemented routes

- **Files:** `lorica-api/openapi.yaml` vs `lorica-api/src/server.rs:215-560`
- 12 endpoints registered in code with no `paths:` entry: `/api/v1/validate/mtls-pem`, `/api/v1/validate/forward-auth`, `/api/v1/dns-providers` (GET/POST), `/api/v1/dns-providers/:id` (PUT/DELETE), `/api/v1/dns-providers/:id/test` (POST), `/api/v1/notifications/history` (GET), `/api/v1/sla/routes/:id/data` (DELETE), `/api/v1/probes/:id/history` (GET), `/api/v1/settings/otel/test` (POST), `/api/v1/logs/export` (GET), `/api/v1/acme/provision-dns-manual/check` (POST), `/api/v1/certificates/:id/renew` (POST). Three accept user-supplied content (`/validate/mtls-pem`, `/validate/forward-auth`, `/settings/otel/test`).

## Lows / Nits (16)

### Defense-in-depth wins (security-side)

- **L-1** - CSP at `lorica-dashboard/src/lib.rs:74` allows `style-src 'unsafe-inline'`. Needed for Svelte CSS-in-JS today; v2.0 candidate to migrate to CSP3 nonces.
- **L-2** - Same CSP: `connect-src 'self' ws:` admits any `ws://` host. Tighten to the explicit management URL.
- **L-3** - Cert-download audit warn (`lorica-api/src/certificates.rs:563-569`) lacks `username` field. Add `Extension(session)` to handler signature, log `username = %session.username`.
- **L-4** - OCSP DER length parsing (`lorica-tls/src/ocsp.rs:115-140`) hand-rolled, can overshoot on `0x82` long-form lengths. Worst case is a fetch-fail, but worth a `fuzz/` harness.
- **L-5** - Notification `webhook_url` not redacted on TOML export (`lorica-config/src/export.rs:53-70`). A Slack webhook with token in path is leaked verbatim. Redact `url`/`webhook_url` keys alongside `smtp_password`.

### Code-quality lows

- **L-6** - `LogStore.insert_waf_event` errors silently swallowed in proxy hot path (`lorica/src/proxy_wiring.rs:2375,3615,3736,3904,3948`: `let _ = ...`). Replace with `if let Err(e) = ... { tracing::warn!(...); }` + Prometheus counter.
- **L-7** - `acme/dns_challengers/cloudflare.rs:48` and `ovh.rs:72` use `reqwest::Client::new()` (no timeout). Use `Client::builder().timeout(30s)` per-provider.
- **L-8** - Frontend download-filename regex (`lorica-dashboard/frontend/src/lib/api.ts:884`) accepts unsanitised characters from server. Post-process: `.replace(/[\x00-\x1f\\/]/g, '_')`.
- **L-9** - `bot_handlers::hex_to_16_bytes(...).expect(...)` (`lorica/src/proxy_wiring/bot_handlers.rs:335`) couples generator + parser by string contract. Move conversion into `fresh_nonce` (`fn fresh_nonce() -> ([u8; 16], String)`).
- **L-10** - Unbounded `tokio::spawn(resolver.resolve_and_cache(ip))` per cache miss (`lorica/src/proxy_wiring.rs:3190-3192`). Under flood, in-flight resolver tasks accumulate. Deduplicate via `dashmap<IpAddr, Arc<OnceCell<...>>>` or cap with `Semaphore::new(256)`.

### Spec / process lows

- **L-11** - Story 7.3 task `[ ] Add dashboard ban list view` unchecked while story `Status: Done`. Frontend has the view, likely a missed checkbox - verify all 4 fields (IP / reason / expiry / unban) and tick.
- **L-12** - Story 7.3 designs `BanEntry` with `BanReason` enum (RateLimit, Flood); code uses `(banned_at, duration_s)` tuple. Dashboard cannot show *why* an IP was banned. Add a `BanReason` enum to the backend and surface it in `/api/v1/bans`.
- **L-13** - Story 7.3 documents fields as `ban_threshold` / `ban_duration_s`; code names them `waf_ban_threshold` / `waf_ban_duration_s`. Update story to match code (code naming is more accurate).
- **L-14** - OpenAPI under-specifies several response schemas with bare `type: object` (e.g. `/api/v1/preferences`). Define proper `components/schemas/Preference`, `Notification` etc. and reference via `$ref`.

### Supply-chain lows

- **L-15** - Maintenance gaps to batch into a `chore(deps)` PR before v1.6.0 release: `hickory-resolver 0.24 -> 0.26` (DNS), `instant-acme 0.7 -> 0.8` (ACME), `x509-parser 0.16 -> 0.18` (cert parsing). Cheap, isolated bumps. Schedule `axum / tower / tower-http` major bumps + `sysinfo 0.33 -> 0.38` for v1.7.0 (breaking).
- **L-16** - `rustls-pemfile 2.x` (RUSTSEC-2025-0134, unmaintained) still transitive via `lorica-tls`. Migrate `lorica-tls` from `rustls-pemfile` to `rustls-pki-types::pem_slice_iter` to close the only outstanding RUSTSEC chain.

## Informational (8)

- **I-1** - Cert-export `delete_orphan` (`lorica-api/src/cert_export.rs:556-660`): five-step defense-in-depth (sanitize -> canonicalize -> parent compare -> `symlink_metadata` reject -> post-canonicalize `starts_with`). Adopt as boilerplate for any future filesystem-write endpoint.
- **I-2** - `validateHttpHeaderValue` (`lorica-dashboard/frontend/src/lib/route-form.test.ts:247-249`) blocks CRLF correctly. CRLF-injection coverage on the dashboard side is solid.

## CI process recommendations (v1.6.0 ride-along)

The drift seen here would have been caught earlier by three small CI checks:

1. **`openapi-drift`** - assert every `.route(` in `lorica-api/src/server.rs` has a matching `openapi.yaml` path. Catches M-14 class issues.
2. **`story-coverage`** - for each `[x]` task in `docs/stories/*.md`, grep the codebase for at least one identifier mentioned in the task body. Catches H-5 class issues.
3. **`doc-numeric-drift`** - extract `Nk`, `N rules`, `default Ns` patterns from `docs/security/*.md` and `README.md`, fail on disagreement. Catches M-15 class issues.

Plus operational additions:
- `cargo deny check` (advisories + sources + bans) in CI - currently advisory tracking is manual via Cargo.toml comments.
- `cargo audit` in `docs/BUMP-CHECKLIST.md` for the release ritual.
- `pnpm audit --prod` in CI - asserts the empty-prod-deps invariant.
- CodeQL via `github/codeql-action/init@v3` with `languages: rust,javascript`, gate PRs on `security-extended`.

Estimated CI engineering: 2-3 days total for the four checks + CodeQL integration.

## Final risk assessment

**No findings are exploitable by an unauthenticated attacker.** H-1 needs a session cookie. H-2 needs an authenticated route with WAF enabled and a request body large enough. H-3 / H-4 reduce protection rather than create a positive vulnerability. H-5 is contract drift, not exploitation.

The cumulative posture is consistent with a security-conscious codebase that has accumulated a v1.5-cycle worth of feature work since the last formal audit (v1.3.0). The natural next step is a focused v1.6.0 hardening epic that closes H-1 through H-5 plus the M-7/M-8/M-9 sync-IO cluster, then resumes feature work with the three CI checks above pinning the regression surface.
