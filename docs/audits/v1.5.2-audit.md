The bulk of remaining findings cluster around two themes : (a) the M-7 LogStore `spawn_blocking` pass left similar patterns un-migrated on `ConfigStore` + `BotStash`, and (b) the management plane has weaker observability than the proxy hot path.

## Critical (3)

### C-1 - Restarted worker silently loses its RPC channel - all subsequent two-phase reloads, RPC rate-limit pushes, RPC verdict cache, RPC breaker queries skip the worker until full-process restart

- **Files:** `lorica-worker/src/manager.rs:360-418` (`restart_worker` returns only `Option<OwnedFd>` via `take_cmd_fd()`, never touches `take_rpc_fd`) ; `lorica/src/main.rs:1589-1716` (crash-handler in `monitor_workers` only re-spawns the legacy command-channel task).
- **Why:** `spawn_worker` creates two socketpairs per worker (legacy `cmd_fd` + pipelined `rpc_fd`). On crash + respawn, only the cmd channel is re-wired. The restarted worker has no entry in `worker_rpc_endpoints` — `coordinate_config_reload` permanently skips it (it never receives Prepare/Commit RPCs), so its `ProxyConfig`, `CertResolver`, and OTel/GeoIP/ASN/bot-secret state stay frozen at boot until the next full process restart. The cert hot-reload work that just shipped in v1.5.2 is silently broken for any worker that has crashed since boot.
- **Verified:** Yes - `restart_worker` signature confirmed `Result<Option<OwnedFd>, WorkerError>`, `take_rpc_fd` exists but is never called from the crash path.
- **Fix:** Make `restart_worker` return `(OwnedFd, OwnedFd)`. Re-spawn the RPC reader task in the crash branch alongside the legacy command-channel task. Synchronously remove the dead worker's stale endpoint from `worker_rpc_endpoints` instead of waiting for the recv loop to EOF.
- **Effort:** `feat`.

### C-2 - Supervisor `reload_rx.recv()` silently swallows broadcast `Lagged` ; per-worker dispatcher misses every config reload that overflowed the 16-slot channel

- **Files:** `lorica/src/main.rs:684` (`reload_bc_tx = broadcast::channel(16)`) ; `:988` (`Ok(seq) = reload_rx.recv() => {...}` inside `tokio::select!`) ; same pattern at `:1622` in the restart-monitor branch.
- **Why:** When `recv()` returns `Err(Lagged(n))`, the `Ok(seq)` pattern does NOT match - tokio discards the value and the branch is silently disabled for this iteration. The worker then misses every reload that fell out of the 16-slot ring while it was busy in another arm of the select (e.g. servicing a long BanIp send-then-recv). The BanIp arm at `:967` explicitly handles `RecvError::Lagged` and exposes `lorica_ban_broadcast_lagged_total` - the reload arm does not. A burst of >16 reloads (TOML import, automated test, GeoIP autoupdate flapping) leaves the worker on a stale config with zero log, zero metric, zero notification.
- **Fix:** Replace with explicit `match` on `Err(Lagged(n))` ; warn + counter + re-issue a single ConfigReload to catch up. Optionally bump channel cap to 1024 to mirror `ban_bc_tx`.
- **Effort:** `patch`.

### C-3 - Dual rate-limit code paths fire on the same request

- **Files:** `lorica/src/proxy_wiring.rs:2647-2700` (new `Route.rate_limit: Option<RateLimit>` engine via `LocalBucket::try_consume`) AND `lorica/src/proxy_wiring.rs:3653-3749` (legacy `Route.rate_limit_rps` / `rate_limit_burst` via `self.rate_limiter.observe`). Model-side at `lorica-config/src/models/route.rs:574-577` (legacy) + `:712` (new).
- **Why:** A route with both legacy fields AND the structured `rate_limit` set goes through TWO independent token-bucket computations and gets blocked by whichever fires first. The two paths use different keys (`{id}|{ip}` vs `{id}:{ip}`) so cross-process state never converges. Auto-ban only wires off the legacy path - an operator migrating to the new `rate_limit` silently loses the auto-ban escalation. Path-rule overrides only know about the legacy fields, so per-path overrides have no effect on routes using the new struct.
- **Fix:** Pick a winner. Either (a) make legacy fields a thin compatibility shim that synthesizes a `RateLimit { capacity: burst.unwrap_or(rps), refill_per_sec: rps, scope: PerIp }` and routes through the new path only, or (b) hard-deprecate the legacy fields with a one-shot DB migration. Then delete the legacy block. Auto-ban needs to be lifted to the unified path.
- **Effort:** `refactor` (with one-shot DB migration if option b).

## High (12)

### H-1 - RUSTSEC-2026-0104 + RUSTSEC-2026-0099 : `rustls-webpki 0.103.12` reachable on every TLS handshake

- **Crate:** `rustls-webpki 0.103.12` ; **Cargo.lock confirmed.**
- **Advisories:** RUSTSEC-2026-0104 (reachable panic on CRL with empty BIT STRING extension, BEFORE signature verification, patched in 0.103.13), RUSTSEC-2026-0099 (permitted-subtree DNS name constraints accepted wildcard certificates, patched in 0.103.13).
- **Why reachable:** Verifier behind `rustls 0.23.37` - both server-side (proxy: `lorica-tls`, mTLS) and client-side (`reqwest`, `lettre`, `instant-acme`, `quinn`). The CRL panic is reachable any time the proxy parses an attacker-controlled CRL on the mTLS path.
- **Fix:** `cargo update -p rustls-webpki --precise 0.103.13`. SemVer-compatible patch, no API surface touched. **Should be the first commit on v1.5.3 / v1.6.0.**
- **Effort:** `patch` (~1 min).

### H-2 - Bot-stash hot path holds tokio::sync::Mutex on shared ConfigStore across synchronous SQLite I/O - same audit M-7 anti-pattern, missed in v1.5.2

- **File:** `lorica/src/bot.rs:202-298` (`insert`, `take`, `captcha_image`, `prune_expired`, `len`).
- **Why:** Every PoW/captcha challenge issuance and every successful verify acquires the cross-worker `tokio::sync::Mutex<ConfigStore>` and runs SQLite synchronously. M-7 fixed this for `LogStore::insert_*` ; `BotEngine` sites were not migrated. Under bot flood (which is exactly when this matters), challenge-stash insert/take rate equals WAF-block rate ; every one serialises with all other ConfigStore consumers. One slow WAL fsync stalls every async task on that mutex.
- **Fix:** Either (a) move bot stash calls into `spawn_blocking` mirroring the M-7 pattern, or (b) split a dedicated `Arc<parking_lot::Mutex<rusqlite::Connection>>` for bot-stash state (separate from `ConfigStore`) - it's ephemeral, can use `synchronous=OFF`, and decouples from the heavy migrations / encryption_key state.
- **Effort:** `refactor`.

### H-3 - ConfigStore async-handler-with-sync-rusqlite : ~80 sites unmigrated by M-7

- **Files:** Dozens across `lorica-api/src/{routes/crud.rs, backends.rs, certificates.rs, settings.rs, sla.rs, dns_providers.rs, probes.rs, loadtest.rs, status.rs, config.rs, auth.rs, acme/*, cert_export.rs, waf.rs, cache.rs}`. Examples : `routes/crud.rs:3195` (`list_routes` then per-route `list_backends_for_route` - N+1 sync calls under one held `tokio::sync::Mutex`) ; same shape in `backends.rs:175`.
- **Why:** M-7 was scoped to `LogStore` ; `ConfigStore` is hit on every state-mutating endpoint AND on the supervisor's two-phase reload coordinator that reads the entire config every reload. Pattern : `state.store.lock().await; let routes = store.list_routes()?; for route in routes { store.list_backends_for_route(...)?; }`. A single operator polling Overview (9 parallel calls every 30s) + auto-refresh on Routes/Logs/Security/SLA piles up reactor stalls.
- **Fix:** Same `tokio::task::spawn_blocking({ let store = Arc::clone(&store); move || { ... } })` pattern as M-7. Several N+1 patterns can fold into `LEFT JOIN ... GROUP BY` SQL to drop the per-row round-trips entirely.
- **Effort:** `refactor` (large surface, 2-3 days).

### H-4 - `health_check_loop` probes backends sequentially ; one slow backend stalls all probes + serial DB writes under tokio mutex

- **File:** `lorica/src/health.rs:104-225`.
- **Why:** Loop is `for backend in &backends { ... http_probe(&url).await ... }`. With 100 backends and one timing out at the 5s `TCP_CONNECT_TIMEOUT`, all other backends' probes are delayed by 5s and the loop's `interval` overruns. A single permanently-down backend silently halves the effective probe rate. All DB writes happen under a re-acquired `tokio::sync::Mutex<ConfigStore>` lock per backend (sync rusqlite call), serialising the whole batch.
- **Fix:** `futures::stream::iter(backends).for_each_concurrent(MAX_PARALLEL_PROBES, ...)` with a sane parallelism cap (e.g. 32). Collect health-status diffs and apply in a single `spawn_blocking` transaction at end-of-round.
- **Effort:** `feat`.

### H-5 - New-route TLS gap : ProxyConfig swapped before cert resolver reload completes (up to 10s OCSP I/O later)

- **File:** `lorica/src/proxy_wiring.rs:1958-2001` (`handle_config_reload_commit`).
- **Why:** Commit handler swaps `ProxyConfig` first (new route live), then reloads cert resolver via `try_fetch_ocsp` (10s per-responder timeout, parallel via `join_all`). During the OCSP window, a TLS handshake to the new route's hostname returns `None` from `CertResolver.resolve()` and the handshake fails. The v1.5.2 followup commit acknowledges this gap exists in single-process mode too but adds neither a metric nor a shorter window. The reply-before-reload reorder solved the supervisor deadline but not the data-plane gap.
- **Fix:** Two complementary moves : (a) reload cert resolver BEFORE swapping ProxyConfig (the 500ms commit deadline will trigger the legacy fallback under slow OCSP, but the legacy handler also reloads the resolver — same shipping order, no data-plane gap) ; (b) move OCSP fetching out of the critical reload path entirely - kick off a background task and have the resolver hot-swap the OCSP staple bytes when each response lands. Per-domain `lorica_cert_resolver_pending_ocsp_seconds` gauge would expose lag. Closes the same gap as #18 (OCSP staple refresh loop).
- **Effort:** `refactor`.

### H-6 - Cert-resolver reload outcome has no Prometheus counter ; failures only surface as a single `warn!`

- **File:** `lorica/src/reload.rs:803-809`.
- **Why:** `reload_cert_resolver` returns `()` ; `Err` arm is a `warn!` and that's it. No `lorica_cert_resolver_reload_total{result}`, no `lorica_cert_resolver_active_domains` gauge. A single malformed cert in the DB makes `cert_resolver.reload()` return `Err` for the whole batch (look at `reload()` in cert_resolver.rs:109-141 - one bad cert short-circuits, leaves the previous map in place). Operator firing a config-save sees "OK" while every TLS handshake to the new cert silently fails.
- **Fix:** Add `lorica_cert_resolver_reload_total{worker_id, result="ok|fail"}` + `lorica_cert_resolver_active_domains{worker_id}`. Make `reload()` partial-tolerant : build cert-by-cert, log+skip the bad ones, swap with the rest.
- **Effort:** `feat`.

### H-7 - `enforce_notification_retention` runs synchronously under `parking_lot::Mutex` from inside the alert dispatcher async loop ; failures are silently swallowed

- **Files:** `lorica/src/main.rs:4081-4086` (caller) ; `lorica-api/src/log_store.rs:652-669` (callee).
- **Why:** After every successful `insert_notification_event`, the dispatcher calls `let _ = store.enforce_notification_retention(500)`. The retention call acquires `self.conn.lock()` (parking_lot, blocks the reactor) and does sync `SELECT COUNT(*) + DELETE LIMIT n`. Same M-7 anti-pattern. Worse, result is silently swallowed - if DELETE fails, the table is unbounded with no log/metric/alert.
- **Fix:** Wrap in `spawn_blocking`, log+meter via `inc_notifier_events_dropped("retention_failed", 1)` (metric exists). Better : trigger retention once-per-N-events instead of per-event so cost amortises.
- **Effort:** `patch`.

### H-8 - `request_filter` is 1805 LOC with ~30 nested early-returns

- **File:** `lorica/src/proxy_wiring.rs:2141-3946`.
- **Why:** Backlog #7 defers the file split ; this is a separately tractable function-level refactor. ACME challenge intercept, bot-protection solve/captcha, global flood, conn cap, IP blocklist, ban list, route lookup, mTLS, two rate-limit paths, GeoIP, bot-protection eval, forward-auth, WAF prefilter, body buffering, host rewrite, redirect, return_status, breaker probing, tracing - all in one async block. Cyclomatic complexity ~70+. Every audit (security, perf, correctness) re-reads all 1805 lines.
- **Fix:** Extract each early-return block into `fn check_<name>(self, session, ctx) -> Result<Option<Decision>>` returning `Some(Decision::Reject(status, reason, body_hint))` or `None`. Top-level becomes a sequence of `if let Some(d) = self.check_X(...)? { return self.write_decision(session, ctx, d).await; }`. Estimated 12 such extractions. `write_decision` collapses the 18 duplicate error-response sites (linked H-9).
- **Effort:** `refactor`.

### H-9 - `run_supervisor` (1363 LOC), `run_worker` (836 LOC), `run_single_process` (555 LOC) duplicate 80% of wiring

- **File:** `lorica/src/main.rs:1184-3938`.
- **Why:** Backlog #8 defers the file split ; this is the function-level duplication. `build_notify_dispatcher`, `spawn_persisted_alert_dispatcher`, `SlaCollector::start_flush_task`, `ProbeScheduler::reload`, `LoadTestEngine::new`, ACME challenge store init, ACME renewal task spawn, cert-expiry check task spawn, basic-auth-cache prune, rate-limit prune, bot-stash prune, GeoIP set_handle, ASN set_handle, rDNS resolver init, blocklist refresh, OTel try-init - all repeat verbatim across 2-3 branches. **The v1.5.2 worker-mode-cert-hotswap bug existed because of this very duplication** (worker branch was missed when supervisor + single were wired). Next bug fix to any of the 15+ shared spawns has the same risk.
- **Fix:** Extract `BackgroundTasks::start(&store, &task_tracker, &alert_sender, role: Role) -> BackgroundTaskHandles` owning the 200-LOC of common spawns. Each `run_*` shrinks to ~150 LOC of role-specific wiring + one call.
- **Effort:** `refactor`.

### H-10 - 18 duplicated "write 4xx/5xx HTML response" sequences in `request_filter`

- **File:** `lorica/src/proxy_wiring.rs` (18 occurrences of the same 7-line pattern : extract_host → render_error_body → ResponseHeader::build → insert Content-Type/Length → write_response_header → write_response_body → return).
- **Why:** Adding a security header (e.g. `X-Content-Type-Options: nosniff`, `Cache-Control: no-store` on 4xx) requires 18 site edits. Two of them already drift : `:3741` adds `Retry-After: 1` + `X-RateLimit-Reset` ; nothing else does even though 503 paths could legitimately advise.
- **Fix:** `async fn write_error_response(session, status, ctx, host, custom_html, reason, extra_headers) -> Result<bool>`. Use in all 18 sites.
- **Effort:** `refactor`.

### H-11 - `update_settings` is 335 LOC of repeated `if let Some(x) = ... { check; settings.x = x }`

- **File:** `lorica-api/src/settings.rs:120-454`.
- **Why:** ~25 fields each repeat the same shape. Bound-checking is inconsistent (some "must be >= 1", some "must be > 0", some "must be in 1..=N", some no upper bound). Adding a setting means hand-rolling 4-7 lines that look like the previous one ; copy-paste drift is the norm.

- **Effort:** `refactor`.

### M-1 - `GET /api/v1/notifications` leaks webhook bearer + Slack URL (TOML scrub asymmetry)

### M-2 - Access-log CSV export vulnerable to spreadsheet formula injection

- **File:** `lorica-api/src/logs.rs:312-318` (`csv_escape`) called from `:418-444`.
- **Why:** `csv_escape` is RFC 4180-compliant for parsing safety but does not prepend a guard (apostrophe / single tab) to fields starting with `=`, `+`, `-`, `@`, `\t`, `\r`. An attacker who controls any persisted access-log field (request `path`, `host`, `error`, IPv6 `client_ip`) plants `=cmd|'/c calc'!A1` or `=HYPERLINK(...)` ; an SOC analyst opening the CSV in Excel/LibreOffice/Sheets executes the formula. CSV downloads have no rate limit either.
- **Fix:** In `csv_escape`, prefix any field whose first char is in `=+-@\t\r` with a leading apostrophe BEFORE the existing quoting check. Add a unit test.
- **Effort:** `patch`.

### M-12 - `RateLimitEngine::try_consume` always allocates `key.to_string()` in `entry()` AND caller does `format!()` per request

- **Files:** `lorica/src/proxy_wiring/engines.rs:243-265` + caller `proxy_wiring.rs:2655-2658`.
- **Why:** Per-route rate-limited request runs `format!("{}|{}", route_id, scope_key)` (alloc 1) then `try_consume(&key, ...)` does `map.entry(key.to_string())` (alloc 2). DashMap can't accept `&str` for entry. At 10k RPS with rate-limit on : 20k string allocations/s/worker.
- **Fix:** `DashMap::get(key)` first (lock-free read on hit, no allocation), fall back to `entry(key.to_string())` on miss. Or change signature to `try_consume(&self, route_id: &str, scope_key: &str, ...)` and join inside only on insert.
- **Effort:** `patch`.

### M-13 - `CertResolver::resolve` allocates `to_lowercase()` + `format!()` for wildcard lookup on every TLS handshake

- **File:** `lorica-tls/src/cert_resolver.rs:151,163`.
- **Why:** Every TLS ClientHello goes through `resolve`. The SNI is `to_lowercase()`'d ; wildcard branch does `format!("*{}", ...)`. At handshake QPS this matters.
- **Fix:** Stack buffer (`[u8; 256]`, DNS names ≤253 chars) + manual ASCII lowercase. For wildcard probe, custom hasher that hashes the wildcard form without materialising it.
- **Effort:** `patch`.

### M-14 - `WafEngine.evaluate` allocates `format!("header:{name}")` per (header, request) and `Vec::new()` per request

- **Files:** `lorica-waf/src/engine/eval.rs:91-100`, `:113-141`.
- **Why:** For every WAF-on request, header loop (5-10 headers) does `format!("header:{name}")` for the field label even when no rule matches. Events `Vec` allocated on every call regardless of outcome.
- **Fix:** `Cow<'static, str>` populated only on a match, or `Field::Path|Query|Header(&str)` enum stringified only on fire. `SmallVec<[WafEvent; 1]>` for events (common case is one).
- **Effort:** `patch`.

### M-15 - `record_request` and friends allocate `status_code.to_string()` per request

- **File:** `lorica-api/src/metrics.rs:161-168` (called from proxy_wiring.rs:5394).

### M-16 - 7 places duplicate `let store = state.store.lock().await; ... drop; spawn_blocking(...).await` pattern from M-7

- **Files:** 7 sites in `lorica-api/src/{logs,waf,settings}.rs` introduced by M-7 fix.

### M-17 - Two-phase commit has no rollback ; partial-commit fleet split goes unobserved

- **File:** `lorica/src/main.rs:2387-2426` (Commit phase loop) + `:2339-2384` (Abort fan-out only fires on Prepare failures).
- **Why:** If 3/4 workers Commit and 1 times out, the supervisor logs `commit_failed` and falls back to `reload_bc_tx_clone.send(seq)` for ALL workers. The 3 already-committed workers re-do the work via legacy ConfigReload (which has H-18 issue : skips apply_*_settings_from_store hooks). Different workers in the same fleet can serve different bot HMAC secrets / GeoIP DBs / OTel endpoints for several seconds.
- **Fix:** When commit_failed is non-empty AND committed is non-empty, log split fleet, increment `lorica_config_reload_split_fleet_total`. Either re-broadcast legacy only to failed workers or accept the split.
- **Effort:** `patch`.

### M-18 - Legacy `CommandType::ConfigReload` worker handler skips `apply_geoip / apply_otel / apply_asn / apply_bot_secret` hooks

### M-19 - `apply_geoip_settings_from_store` / `apply_asn_settings_from_store` swallow store errors silently
