### Strengths worth calling out

- Every `unsafe` block has a `SAFETY:` comment stating owner/lifetime/
  promotion preconditions (lorica-shmem, fd_passing, RpcEndpoint).
- `rustls-pemfile = "=2.1.2"` hard-pinned per RUSTSEC-2025-0134.
- `deny.toml` rejects unknown registries and unknown git sources; tight
  license allowlist.
- Frontend ships zero runtime npm dependencies; pnpm lockfile with SHA-512
  integrity.
- `lorica-worker` and `lorica-shmem` (privileged surface) each have only 4
  direct deps.
- Argon2id with OWASP-recommended params (19 MiB, 2 iters, v0x13).
- No SQL injection surface (all queries parameterised through
  `ConfigStore`).
- Session cookies: `HttpOnly; Secure; SameSite=Strict; Path=/api`.
- Pipelined RPC has documented cancel-safety + 200-way fan-out tests.
- `GenerationGate` CAS-loop proven by concurrent race test.
- Fail-open semantics on transport error (breaker + verdict cache both
  degrade gracefully so a flaky UDS channel cannot DoS the data plane).

### Headline findings

- **[H-1]** Supervisor breaker deadlocks a backend if the probe-admitted
  worker crashes between admit and report (`lorica/src/main.rs:1820-1829`).
  Locks the backend out until supervisor restart.
- **[H-2]** `recv_worker_fds` leaks received kernel FDs on error paths
  (UTF-8 fail, fds/tokens mismatch, bad `FdKind` token). Also silently
  truncates on `PAYLOAD_BUF_SIZE` overflow (no `MSG_TRUNC` check).
- **[H-3]** Two-phase reload commit re-reads the connection filter CIDRs
  instead of using the `PreparedReload` snapshot. ProxyConfig and filter
  briefly desync on a reload that touches both. Not a security hole, but
  violates the two-phase atomicity intent.

None of these block v1.3.0 functionally - they are edge-case reliability and
lifecycle issues, not security vulnerabilities. See **Recommendation** at the
end.

## 4. Rust Code Review

**Verdict: concerns (non-blocking) - ready to tag after addressing H-1/H-2/H-3.**

### 4.1 Highs (3)

- **[H-1] Supervisor breaker deadlocks on crashed probe worker**
  - File: `lorica/src/main.rs:1820-1829` (`SupervisorBreakerRegistry::query`)
  - State `HalfOpen { probe_in_flight: false }` flips to `probe_in_flight: true`
    and returns `AllowProbe`. The only reset path is a subsequent `report()`.
    If the worker dies, crashes, or times out mid-probe, the breaker entry
    is stuck; every subsequent query returns `Deny` until supervisor restart.
  - Remediation: add `probe_started_at: Instant` to `HalfOpen`; if
    `probe_started_at.elapsed() > cooldown` on a fresh query, treat as failed
    probe (state -> Open, failures += 1). Alternative: supervisor sweep task
    synthesising `BreakerReport { success: false, was_probe: true }` for
    entries stuck > `cooldown * 2`.

- **[H-2] `recv_worker_fds` leaks received FDs on error paths**
  - File: `lorica-worker/src/fd_passing.rs:131-164`
  - `recvmsg` returns raw `RawFd`s; on UTF-8 validation failure, fds/tokens
    count mismatch, or any bad `FdKind` token, the function returns `Err`
    without closing the kernel FDs. `PAYLOAD_BUF_SIZE = 2048` silently
    truncates (no `MSG_TRUNC` check).
  - Remediation: wrap each received `RawFd` into `OwnedFd` immediately after
    `recvmsg`; map back via `IntoRawFd` on success. Check
    `msg.flags.contains(MsgFlags::MSG_TRUNC)` and return
    `WorkerError::InvalidPayload` on truncation.

- **[H-3] Two-phase reload commit re-reads connection filter CIDRs**
  - Files: `lorica/src/proxy_wiring.rs:1917-1930` + `lorica/src/reload.rs:88`
  - `PendingProxyConfig` only holds `Arc<ProxyConfig>`, not the full
    `PreparedReload`. TODO at 1923-1930 documents the gap. On a reload that
    also flips `connection_allow_cidrs`/`connection_deny_cidrs`, there is a
    brief window where the new `ProxyConfig` is published but the connection
    filter still reflects the old CIDRs.
  - Remediation: change `PendingProxyConfig` to hold `Arc<PreparedReload>`;
    publish ProxyConfig ArcSwap AND reload the connection filter from the
    same Commit handler. Drop the double Arc wrap in
    `commit_prepared_reload`.

## 6. Recommendation

Three paths:

**A. Fix H-1/H-2/H-3 before tagging v1.3.0** (~1 day of focused work)
- Thread `PreparedReload` through two-phase commit (H-3).
- `OwnedFd` wrap + `MSG_TRUNC` check in `recv_worker_fds` (H-2).
- Probe timeout in `SupervisorBreakerRegistry::query` (H-1).
- Adds three commits; regenerate docker e2e.
- Cleanest ship.

**B. Tag v1.3.0 now, fix all three in a v1.3.1 patch release** (same-week)
- Add a "Known issues" section to the v1.3.0 release notes mentioning the
  three items by severity + workaround.
- Track as v1.3.1 milestone issues.
- Viable because none are security-critical and all three are lifecycle
  edge cases (crashed probe worker, fd leak on worker init error, reload
  with CIDR change).

**C. Ship v1.3.0, address in v1.4.0 roadmap**
- Not recommended: H-1 is a real reliability bug that can bite in
  production with misbehaving workers.

**My recommendation: Option A.** The fixes are local, high-confidence, and
the v1.3.0 changelog explicitly advertises cross-worker breaker parity as a
feature - shipping it with H-1 present would undermine the guarantee.

If that's not practical, Option B is acceptable. Option C is not.
